# Registration Module — iam-operations

**Module Path:** `iam-operations/src/main/java/com/nihith/iam`

This document describes the data structures, interfaces, and implementations supporting the **User Registration** flow in Breakdown. Registration is the first step in Breakdown's authentication lifecycle; users provide a username and password, which are validated and persisted.

## Domain Model: RegistrationRequest

**File:** `src/main/java/com/nihith/iam/model/RegistrationRequest.java`

The `RegistrationRequest` POJO encapsulates the minimal user information needed to register a new user. It embodies Breakdown's **minimal-PII philosophy**—only username and password are required from the user.

### Fields

| Field | Type | Validation | Purpose |
|-------|------|-----------|---------|
| `username` | `String` | `@NotEmpty`, `@Size(min=3, max=50)` | Unique identifier chosen by user at signup |
| `password` | `String` | `@NotEmpty`, `@Size(min=8, max=255)` | User's login secret; hashed before storage |
| `platform` | `String` | — | OS/client type (e.g., `"iOS"`, `"Android"`, `"Web"`); enriched by controller from HTTP headers |
| `userAgent` | `String` | — | User's browser/client agent string; enriched by controller from HTTP headers |
| `ipAddress` | `String` | — | Client IP; enriched by controller from HTTP headers |
| `userId` | `String` | — | Server-generated UUID; populated by service layer before persistence |

### Constructors

```java
// No-arg constructor required for Jackson deserialization
public RegistrationRequest()

// Convenience constructor for tests
public RegistrationRequest(String username, String password)
```

### Getters and Setters

Standard JavaBean style: `getField()` and `setField(value)`. No Lombok.

### Key Design Decisions

1. **Minimal PII:** Only `username` and `password` are collected from the user request. Device/location metadata (`platform`, `userAgent`, `ipAddress`) are extracted from HTTP headers by the REST controller and attached to the request object before delegating to the service layer.

2. **No Email Field:** Breakdown does not store email addresses at registration. This reduces the platform's attack surface and compliance burden (GDPR, CCPA).

3. **Password Validation Constraints:**
   - Minimum 8 characters (industry standard for password strength)
   - Maximum 255 characters (matches typical hashed password storage limits)
   - Constraints are enforced via `jakarta.validation` annotations, not service-level checks

4. **UUID Generation:** The service layer generates a server-side UUID (`userId`) before creation; clients never provide a user ID.

5. **JSON Deserialization:** Annotated with `@JsonIgnoreProperties(ignoreUnknown = true)` to tolerate extra fields in incoming JSON (forwards compatibility).

### Usage in Registration Flow

```
User → REST Controller (validates, enriches platform/userAgent/ipAddress)
    ↓
RegistrationRequest instance created
    ↓
IAMDelegator.createUser(RegistrationRequest)
    ↓
UserIAMService implementation (MongoDB or Kratos)
```

## Interface Contract: UserIAMService

**File:** `src/main/java/com/nihith/iam/interfaces/UserIAMService.java`

`UserIAMService` is the abstraction layer that allows Breakdown to support multiple user-identity backends:
- **MongoDB** (Phase 1): Direct access to a MongoDB collection
- **Ory Kratos** (Phase 2): REST delegation to an external identity provider

All methods throw `IAMException` for both data-store and network errors, giving controllers a single error type to handle.

### Methods

#### `createUser(User user)`

```java
/**
 * Persists a new user record.
 *
 * @param user the user to create; must carry a hashed password (never plaintext)
 * @return {@code true} if the creation was successful, {@code false} otherwise
 * @throws IAMException if a data store or upstream error occurs
 */
boolean createUser(User user) throws IAMException;
```

**Responsibility:** Insert a new user into the backing store (MongoDB collection or Kratos instance).

**Input Contract:**
- `user` must be fully initialized with:
  - `userId`: UUID generated by caller
  - `username`: validated, unique username
  - `passwordHash`: bcrypt or equivalent (never plaintext password)
  - Optional: `platform`, `userAgent`, `ipAddress`, roles

**Output Contract:**
- `true` if the user was successfully created
- `false` if creation failed (collision on unique index, validation failure, etc.)

**Error Handling:** Wrap all infrastructure errors (MongoDB connection failures, Kratos API timeouts, etc.) as `IAMException`.

---

#### `findByUsername(String username)`

```java
/**
 * Retrieves a user by their unique username. This is the primary lookup used
 * by POST /auth/login — the userId field of the login request
 * holds the human-readable username the user chose at registration.
 *
 * @param username the username to look up
 * @return the matching {@link User}, or {@code null} if not found
 * @throws IAMException if a data store or upstream error occurs
 */
User findByUsername(String username) throws IAMException;
```

**Responsibility:** Retrieve a single user record by username.

**Usage:** The Login flow calls this to fetch user credentials for password validation.

**Output Contract:**
- Non-null `User` if exactly one match found
- `null` if no matches found
- Throws `IAMException` for infrastructure errors

---

#### `findByUserId(String userId)`

```java
/**
 * Retrieves a user by their unique user ID. Used by /auth/refresh
 * to validate that the user still exists when refreshing a token.
 *
 * @param userId the user ID to look up
 * @return the matching {@link User}, or {@code null} if not found
 * @throws IAMException if a data store or upstream error occurs
 */
User findByUserId(String userId) throws IAMException;
```

**Responsibility:** Retrieve a user by their server-assigned UUID.

**Usage:** Token refresh validates that the user still exists (not deleted).

---

#### `usernameExists(String username)`

```java
/**
 * Checks whether a username already exists in the user store.
 * Used during registration to detect duplicate usernames and prevent
 * enumeration attacks by returning a generic error.
 *
 * @param username the username to check
 * @return {@code true} if username exists, {@code false} if available
 * @throws IAMException if a data store error occurs
 */
boolean usernameExists(String username) throws IAMException;
```

**Responsibility:** Perform a lightweight check for username availability without retrieving the full user record.

**Usage:** Registration flow checks this before calling `createUser()`; if `true`, returns a generic "registration failed" message to avoid enumeration attacks.

## MongoDB Implementation: UserMongoDao

**File:** `src/main/java/com/nihith/iam/dao/UserMongoDao.java`

`UserMongoDao` implements `UserIAMService` using direct MongoDB operations via `MongoDBOperations`.

### Class Declaration

```java
@Component
@Scope(value = BeanDefinition.SCOPE_SINGLETON)
public class UserMongoDao implements UserIAMService {

    private static final Logger logger = LogManager.getLogger(UserMongoDao.class);
    public static final String USER_COLLECTION_NAME = "users";

    @Autowired
    private MongoDBOperations mongoDBOperations;
```

### Collection

**Name:** `"users"`

**Indexes:**
- Unique index on `username` (prevents duplicate registrations)
- Index on `userId` (for token refresh lookups)

### Method Implementation: `usernameExists(String username)`

This is the key method for registration:

```java
@Override
public boolean usernameExists(String username) throws IAMException {
    logger.info("Entered usernameExists");
    try {
        List<User> users = mongoDBOperations.fetchRecordsWithFilter(
                mongoDBOperations.getCollection(USER_COLLECTION_NAME),
                UserQueryBuilder.filterByUsername(username),
                User.class);
        logger.info("Exiting usernameExists");
        return !users.isEmpty();
    } catch (MongoException e) {
        throwIAMException(e);
        return false;
    }
}
```

**Flow:**
1. Query the `users` collection with filter `{username: "<provided>"}` via `UserQueryBuilder.filterByUsername(username)`
2. If result list is non-empty, username exists → return `true`
3. If result list is empty, username is available → return `false`
4. If `MongoException` occurs, log and rethrow as `IAMException`

**Query Used:**
```java
// UserQueryBuilder.filterByUsername(username)
BasicDBObject filter = new BasicDBObject();
filter.put("username", username);
return filter;
```

### Other Methods

- **`createUser(User user)`** → Serializes to BSON `Document`, calls `mongoDBOperations.insertSingleDocument()`
- **`findByUsername(String username)`** → Calls `fetchRecordsWithFilter()` with same `UserQueryBuilder.filterByUsername()` query
- **`findByUserId(String userId)`** → Calls `fetchRecordsWithFilter()` with `UserQueryBuilder.filterByUserId()` query

### Error Handling

All MongoDB exceptions are caught and wrapped as `IAMException` via a shared helper:

```java
private void throwIAMException(MongoException e) {
    logger.error("Mongo Exception Occurred");
    logger.error(ExceptionUtils.getStackTrace(e));
    throw new IAMException("Mongo Exception Occurred");
}
```

## Kratos Implementation: KratosUserIAMService

**File:** `src/main/java/com/nihith/iam/dao/KratosUserIAMService.java`

`KratosUserIAMService` is a **Phase 2 stub** that delegates user-identity operations to an **Ory Kratos** instance via REST APIs.

### Current Status

All methods currently throw `IAMException` with a deferred message:

```java
@Override
public boolean usernameExists(String username) throws IAMException {
    logger.info("Entered usernameExists (Kratos) — not yet implemented");
    throw new IAMException("Kratos registration support deferred to Phase 2");
}
```

### Environment Variables

The class reads these at initialization:

| Variable | Purpose | Example |
|----------|---------|---------|
| `KRATOS_ADMIN_URL` | Base URL of Kratos Admin API (write operations) | `http://kratos:4434` |
| `KRATOS_PUBLIC_URL` | Base URL of Kratos Public API (login flows) | `http://kratos:4433` |

Both are read via `@PostConstruct`:

```java
@PostConstruct
public void init() {
    this.kratosAdminUrl = EnvironmentUtil.getEnvironmentVariable(KRATOS_ADMIN_URL);
    this.kratosPublicUrl = EnvironmentUtil.getEnvironmentVariable(KRATOS_PUBLIC_URL);
    logger.info("Kratos Admin URL::{}", this.kratosAdminUrl);
    logger.info("Kratos Public URL::{}", this.kratosPublicUrl);
}
```

### Future Implementation (Phase 2)

When Phase 2 begins, methods will:
1. Construct HTTP requests to Kratos Admin API endpoints
2. Serialize/deserialize JSON request/response bodies
3. Handle HTTP status codes and Kratos error responses
4. Wrap infrastructure errors as `IAMException`

Example pseudocode for Phase 2 `createUser()`:

```java
// POST /admin/identities (Kratos Admin API)
POST http://kratos:4434/admin/identities
{
  "schema_id": "default",
  "credentials": {
    "password": { "config": { "password": "bcrypt_hash" } }
  },
  "traits": {
    "username": "alice"
  }
}
```

### Note on Token Storage

`KratosUserIAMService` handles **user identity only** (registration, login). Token storage always uses MongoDB via `TokenMongoDao` — the `PREFERRED_IAM=kratos` environment variable does **not** affect token operations.

## Constants: AuthMessageConstants

**File:** `src/main/java/com/nihith/iam/constants/AuthMessageConstants.java`

String constants used in response messages. These are returned to clients to communicate success/failure of auth operations.

### Error Messages

| Constant | Value | Usage |
|----------|-------|-------|
| `INTERNAL_SERVER_ERROR` | `"Internal Server Error"` | Catch-all for unexpected errors |
| `INVALID_CREDENTIALS` | `"Invalid username or password"` | Login failure (generic, to prevent enumeration) |
| `USER_CREATION_FAILURE` | `"User Creation Failed"` | Registration creation error |
| `TOKEN_CREATION_FAILURE` | `"Token Creation Failed"` | Token generation failure |
| `LOGIN_FAILURE` | `"Login Failed"` | Generic login error |
| `INVALID_REFRESH_TOKEN` | `"Invalid or expired refresh token"` | Token refresh error |

### Success Messages

| Constant | Value | Usage |
|----------|-------|-------|
| `USER_CREATION_SUCCESS` | `"User Registered Successfully"` | Registration success |
| `LOGIN_SUCCESS` | `"Login Successful"` | Login success |
| `LOGOUT_SUCCESS` | `"Logged Out Successfully"` | Logout success |
| `REGISTRATION_SUCCESS` | `"User Registered Successfully"` | Alias for `USER_CREATION_SUCCESS` |

### Registration-Specific Messages

| Constant | Value | Usage |
|----------|-------|-------|
| `REGISTRATION_INVALID_DATA` | `"Unable to complete registration. Please check your username and password and try again."` | Validation failure or duplicate username (generic) |

### Design: Generic Messages

Notice that error messages are intentionally generic (e.g., `"Invalid username or password"` rather than `"Username does not exist"`). This prevents **enumeration attacks** where attackers use error messages to discover which usernames are registered on the platform.

## Privacy & Security Rationale

### Minimal PII Collection

Breakdown collects only **username and password** at registration:

- **No email:** Reduces attack surface, eliminates email verification workflows, reduces GDPR/CCPA compliance burden
- **No phone number:** Avoids SMS/OTP complexity; out-of-scope for MVP
- **No personal details:** Name, address, date of birth, etc. are never collected during registration

### Device Metadata

`platform`, `userAgent`, and `ipAddress` are **enriched by the REST controller** from HTTP request headers, not collected from user input. This serves audit/forensics purposes without requiring user cooperation.

### Generic Error Messages

Registration failures return a single generic message: `"Unable to complete registration. Please check your username and password and try again."` This prevents **username enumeration attacks**:

- Attacker cannot distinguish between "username already taken" and "invalid password format"
- Attacker cannot build a list of registered usernames by brute-forcing the registration endpoint

Similarly, login returns `"Invalid username or password"` (not `"User not found"`) to prevent enumeration of valid usernames.

### Password Hashing

- Passwords are **never stored in plaintext**
- Service layer hashes passwords using bcrypt (or equivalent) before calling `UserIAMService.createUser()`
- `RegistrationRequest` carries plaintext password (for validation); service layer performs hashing before persistence

### UUID User IDs

- Usernames are **human-readable** (chosen by user, used for login)
- User IDs are **server-generated UUIDs** (unpredictable, used in tokens and URLs)
- This separation prevents **user enumeration** via sequential integer IDs (old insecure pattern)

### Kratos Phase 2 Compliance

When Phase 2 integrates Ory Kratos:
- Kratos will handle password hashing and storage (Breakdown delegates this to the provider)
- Breakdown will continue to wrap Kratos APIs with the same generic error messages
- Token storage remains in MongoDB (not Kratos)
